## 0. Cross-cutting rulings (read first ‚Äî these bind several rows below)

1. **ADR-011 is the Realtime Backplane (Redis pub/sub + Streams).** Three docs already hardcode
   `../adr/ADR-011-realtime-backplane.md` as a link target (ARCHITECTURE.md, REALTIME.md, DEPLOYMENT.md)
   and the punch-list B2 explicitly assigns **D-011** to it. Therefore the backplane takes **ADR-011 / D-011**.
   The "process-discipline ADR" floated in the decision-ledger Open Questions is **renumbered to ADR-012 / D-012
   and deferred** (see PROC-1). No file currently links to a process-discipline ADR-011, so there is no broken link.
2. **New-ADR filenames must match the link targets the docs already use** (long, descriptive form):
   `adr/ADR-009-minio-storage.md`, `adr/ADR-010-docker-first.md`, `adr/ADR-011-realtime-backplane.md`.
   (Note: the eight existing ADRs on disk use the short form `ADR-00N-<word>.md` while the docs/ledger link to the
   long form. That filename-vs-link skew is pre-existing and is logged as **DOC-1 / Deferred-to-Phase-1**, not fixed here.)
3. **Adding a collection, field, index, or realtime event is a data/contract-model change, not an architecture
   decision** ‚Äî it travels with a history entry + context (canon) update (R3/R4), and does **not** require its own ADR
   (B3, B4, B5, B6). Only B1 and B2 produce ADRs, because only they ratify/elevate an architecture-level decision.
4. **Repomix regeneration is `pending first code`.** No source exists yet, so repomix is not run; record the pending
   note in the decision-ledger / a repomix README rather than executing repomix (per the punch-list PROCESS rule).
5. **Default stance:** adopt the recommendation already noted in each doc's Open Questions unless flawed. Every
   item below adopts its source recommendation; none were found flawed except the ADR-011 numbering collision (resolved in #1).
